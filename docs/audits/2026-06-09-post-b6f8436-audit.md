## Findings

### P1: V2 recording output is polluted by a V1 lap-index append

- Locations:
  - `src/recording/controller.rs:194`
  - `src/bin/acc-live-telemetry.rs:172`
  - `src/bin/acc-live-telemetry.rs:264`
  - `src/bin/acc-live-telemetry.rs:859`
  - `src/bin/acc-live-telemetry.rs:976`
  - `src/recording/outcome.rs:411`
- Symptom:
  - `BinaryTelemetryWriterV2::finish()` already writes the V2 footer and V2 lap entries.
  - After that, `RecordingController`, `record`, and `record-all` call `append_lap_index()`.
  - `append_lap_index()` appends the legacy V1 `LAP_INDEX_MAGIC` block directly to the file tail.
- Impact:
  - `.acctlm2` files get trailing V1-only data after the V2 footer.
  - The V2 header/footer no longer describe the true logical end of the file.
  - Strict V2 readers, validators, or future tools may treat the file as malformed or containing trailing garbage.
  - `summary.total_bytes` and file-size reporting include bytes that are not part of the V2 format.

#### Repair options and impact

Safe local fix:

- Make `append_lap_index()` a V1-only operation.
- Detect file format before appending:
  - If the file magic is V2 (`ACT2`), return `Ok(0)` without writing.
  - If the file is V1, preserve the current append behavior.

Alternative:

- Split the function into `append_lap_index_v1()` and a separate V2-aware path.
- Update CLI/controller call sites to call the V1 function only when writing or repairing V1 files.

Expected impact:

- Low.
- Normal V2 recording should become more format-correct.
- V1 files keep the existing behavior.
- Only consumers that accidentally depend on a V1 lap-index block at the end of `.acctlm2` files would be affected; that should be treated as reliance on a bug, not a supported contract.

Recommended tests:

- Record/write a V2 file, finish it, run the ending path, and assert no V1 `LAP_INDEX_MAGIC` appears after the V2 footer.
- Keep a V1 append regression test to prove the legacy path still works.

### P1: V1-to-V2 converter reports success even when verification detects data loss

- Location:
  - `src/bin/acctlm-to-acctlm2.rs:165`
- Symptom:
  - The converter verifies the output after writing.
  - If frame counts differ, it prints a warning and continues.
  - If checked fields mismatch, it prints a warning and still returns `Ok(())`.
- Impact:
  - Automation and CI see a successful exit code for a failed "lossless" conversion.
  - A corrupted or incomplete `.acctlm2` file can be published or consumed downstream.
  - The converter's documented lossless behavior is not enforced.

#### Repair options and impact

Safe local fix:

- Turn verification mismatches into errors:
  - Frame count mismatch should return `TelemetryError::InvalidFormat`.
  - Any checked field mismatch should return `TelemetryError::InvalidFormat`.

Optional compatibility switch:

- Add `--allow-mismatch` for manual diagnostic use.
- Default behavior should remain strict.

Expected impact:

- Low to medium.
- Successful conversions are unaffected.
- Existing scripts that currently ignore converter warnings will begin failing when data does not roundtrip.
- This is a behavior tightening, but it matches the converter's lossless contract.

Recommended tests:

- Add a converter test where verification detects a mismatch and asserts a non-zero failure path.
- Keep the existing successful conversion roundtrip tests.

### P2: Stopping before ACC reaches Live is surfaced as an error

- Locations:
  - `src/recording/engine.rs:121`
  - `src/recording/engine.rs:129`
  - `src/recording/controller.rs:228`
- Symptom:
  - `RecordingController::start()` sends `RecordingStarted` once the thread starts, even if the engine is still waiting for ACC to become Live.
  - If `stop()` is called before the writer is created, `run_recording_loop()` returns `InvalidArgument("stop requested before recording started")`.
  - The controller converts that into `RecordingStatus::Error` and does not send an outcome.
- Impact:
  - A user-initiated cancel while waiting for Live is reported as a recording error.
  - UI callers that wait only for `RecordingOutcome` may hang or time out.
  - API lifecycle semantics are unclear: `stop()` is documented as a normal way to end recording, but one valid timing window is treated as invalid.

#### Repair options and impact

Minimal local fix:

- Treat stop-before-start as a normal cancellation instead of `TelemetryError`.
- Emit a status such as `Stopping { reason: StopReason::Manual }` or introduce a stopped/cancelled status.
- Do not emit `RecordingStatus::Error` for this case.

More complete API fix:

- Extend the outcome channel to distinguish:
  - completed recording with a file
  - cancelled before any file was created
  - failed recording
- This could be done with an enum such as `RecordingCompletion::{Finished(RecordingOutcome), Cancelled, Failed(...)}`.

Expected impact:

- Minimal fix: low to medium.
  - Existing outcome semantics stay intact.
  - Callers must listen to status to observe cancellation before any file exists.
- Complete API fix: medium.
  - More explicit and robust, but requires downstream callers to handle a new completion type.

Recommended tests:

- Start a scripted source that remains `Off` or `Unavailable`.
- Call `stop()` before any Live frame.
- Assert that status reports cancellation/manual stop, not error.
- Assert no output file is created.

### P2: V2 reader can hang or panic on malformed row-group/column metadata

- Locations:
  - `src/reader_v2.rs:1178`
  - `src/reader_v2.rs:1299`
  - `src/reader_v2.rs:1326`
  - `src/reader_v2.rs:1350`
- Symptom:
  - `scan_row_groups()` advances `pos` to `rg_end`.
  - For a malformed row group with valid `RGHD` magic but no valid group end, `rg_end` can remain equal to `pos`, causing an infinite loop.
  - Column reads slice with `col_data_offset..col_data_offset + byte_len` without checking the end against `group_bytes.len()`.
- Impact:
  - Corrupt or malicious `.acctlm2` files can cause hangs or panic instead of returning `TelemetryError::InvalidFormat`.
  - This affects CLI operations that open or inspect user-provided files.
  - The risk is not covered by the current default test run because many V2 edge-case tests remain ignored.

#### Repair options and impact

Safe local fix:

- In `scan_row_groups()`:
  - Validate that row-group parsing makes forward progress.
  - Return `InvalidFormat` if the computed next offset is less than or equal to the current offset.
  - Validate each `group_end` against `data.len()` before trusting it.
- In column decode paths:
  - Check `col_data_offset + entry.byte_len as usize <= group_bytes.len()` before slicing.
  - Return `TelemetryError::InvalidFormat` on overflow or out-of-range columns.
  - Use checked arithmetic for offset additions.

Expected impact:

- Low.
- Valid V2 files should read exactly as before.
- Invalid files change from hang/panic behavior to structured errors.
- This makes CLI behavior safer without changing normal APIs.

Recommended tests:

- Enable or rewrite targeted cases from `tests/v2_edge_cases.rs`.
- Add tests for:
  - zero-progress row group
  - truncated row-group header
  - column byte length extending past group end
  - invalid skip-index offset

### S1: `flush_interval` 配置未在 recording engine 中使用

- Locations:
  - `src/recording/engine.rs`
  - `src/recording/controller.rs:115`
- Symptom:
  - `RecordingEngineConfig` 定义了 `flush_interval: Option<Duration>` 字段。
  - `controller.rs:115` 设置了 `flush_interval: Some(Duration::from_secs(2))`。
  - 但 `run_recording_loop()` 的主循环中完全没有检查或使用该配置，writer 的 `flush()` 从未被主动调用。
- Impact:
  - 该配置项被静默忽略，对运行时行为没有任何影响。
  - 数据仅在 `writer.finish()`（录制结束）时 flush 到磁盘。
  - 如果进程崩溃或系统断电，录制期间的所有数据可能丢失。
  - 用户配置的 flush 间隔没有起到预期的数据保护作用。

#### Repair options and impact

Safe local fix:

- 在 `run_recording_loop()` 的主循环中加入 `flush_interval` 检查：
  - 在 `RecordingEngineConfig` 旁边维护一个 `last_flush: Instant`。
  - 每次循环 tick 检查 `last_flush.elapsed() >= flush_interval`。
  - 如果条件满足且 `writer` 存在，调用 `writer.flush()` 并更新 `last_flush`。
- 如果 `flush_interval` 为 `None`，保持当前行为（仅在 finish 时 flush）。

Expected impact:

- Low.
- 正常录制行为不变，只是增加了磁盘写入频率。
- 崩溃时的数据丢失窗口从"整个录制期间"缩小到"最多一个 flush_interval"。

Recommended tests:

- 设置 `flush_interval = Some(Duration::from_millis(100))`，录制若干帧后强制终止进程，验证文件已部分写入。
- 设置 `flush_interval = None`，验证仅在 finish 时写入。

### S2: V2 writer 最后一圈 lap entry 的 validity 状态被硬编码

- Location:
  - `src/writer_v2.rs:339-345`
- Symptom:
  - `BinaryTelemetryWriterV2::finish()` 中，如果 `current_lap` 存在，追加的 `LapIndexEntryV2` 总是硬编码 `is_valid: 1, is_out_lap: 0`。
  - 这与 `engine.rs` 中正确的 lap validity 检测逻辑不一致（engine 会检测 `is_valid_lap != 0` 和 out-lap 状态）。
- Impact:
  - V2 footer 中的 lap metadata 不准确。
  - 读取 V2 lap index 的工具可能得到错误的 validity / out-lap 信息。
  - 如果 future tools 依赖 V2 footer 的 lap metadata 而不是重新扫描文件，会得到错误结果。

#### Repair options and impact

Safe local fix:

- 在 `BinaryTelemetryWriterV2` 中增加字段跟踪当前圈的 `is_valid: bool` 和 `is_out_lap: bool`。
- 在 `write_frame()` 中根据 session 数据更新这些状态。
- 在 `finish()` 中使用实际状态而不是硬编码。

Expected impact:

- Low.
- 只有 V2 footer 中的 lap metadata 会改变，实际数据不受影响。
- 需要确认是否有下游工具依赖当前硬编码的行为（应该没有）。

Recommended tests:

- 录制包含 out-lap 和 invalid lap 的数据，验证 V2 footer 中的 lap metadata 与重新扫描文件的结果一致。

### S3: `append_lap_index` 读取整个 session cluster 可能导致 OOM

- Location:
  - `src/recording/outcome.rs:363`
- Symptom:
  - `append_lap_index()` 调用 `BinaryTelemetryReader::open(path)` 然后 `read_all_session()`。
  - `read_all_session()` 会将整个文件的 session cluster 全部读入内存。
- Impact:
  - 对于长时间录制的文件（如数小时的 endurance race），session cluster 可能非常大。
  - 可能导致内存占用过高甚至 OOM。
  - 由于 `append_lap_index` 在录制结束后调用，此时用户可能正在进行其他操作，突然的内存压力会影响用户体验。

#### Repair options and impact

Safe local fix:

- 使用流式读取代替 `read_all_session()`：
  - 只读取 `sample_tick` 和 `normalized_car_position` 两个字段。
  - 逐 chunk 读取，而不是一次性读入内存。
- 或者，在 `BinaryTelemetryReader` 上增加一个只读取指定字段的 API。

Alternative:

- 完全避免 `append_lap_index` 对 V2 文件的调用（修复 P1 后这个问题自动消失）。
- 对于仍然需要 V1 lap-index 的场景，可以限制文件大小或增加内存警告。

Expected impact:

- Low.
- 正常大小的文件不受影响。
- 大型文件的内存占用会显著降低。

Recommended tests:

- 用一个超大文件测试 `append_lap_index()` 的内存占用。
- 验证流式读取的 lap boundary 检测结果与 `read_all_session()` 一致。

## Suggested Fix Order

1. Stop appending V1 lap-index data to V2 files.

5. Implement `flush_interval` handling in the recording engine.
6. Fix V2 writer hard-coded lap validity in footer metadata.
7. Optimize `append_lap_index` memory usage for large files (or remove for V2).
