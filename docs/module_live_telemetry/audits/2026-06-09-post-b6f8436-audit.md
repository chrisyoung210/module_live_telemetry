### P1：V2 录制输出被 V1 圈索引追加所污染

- 位置：
  - `src/recording/controller.rs:194`
  - `src/bin/acc-live-telemetry.rs:172`
  - `src/bin/acc-live-telemetry.rs:264`
  - `src/bin/acc-live-telemetry.rs:859`
  - `src/bin/acc-live-telemetry.rs:976`
  - `src/recording/outcome.rs:411`
- 症状：
  - `BinaryTelemetryWriterV2::finish()` 已经写入了 V2 footer 和 V2 圈条目。
  - 在此之后，`RecordingController`、`record` 和 `record-all` 调用 `append_lap_index()`。
  - `append_lap_index()` 将遗留的 V1 `LAP_INDEX_MAGIC` 块直接追加到文件末尾。
- 影响：
  - `.acctlm2` 文件在 V2 footer 之后会带有 V1 专属的尾部数据。
  - V2 header/footer 不再描述文件真正的逻辑结尾。
  - 严格的 V2 读取器、验证器或未来工具可能将文件视为格式错误或包含尾部垃圾数据。
  - `summary.total_bytes` 和文件大小报告包含了不属于 V2 格式的字节。

#### 修复方案及影响

安全的本地修复：

- 将 `append_lap_index()` 改为仅 V1 操作。
- 在追加前检测文件格式：
  - 如果文件魔数是 V2（`ACT2`），返回 `Ok(0)` 而不写入。
  - 如果文件是 V1，保留当前的追加行为。

替代方案：

- 将函数拆分为 `append_lap_index_v1()` 和一个独立的 V2 感知路径。
- 更新 CLI/控制器调用点，使其仅在写入或修复 V1 文件时调用 V1 函数。

预期影响：

- 低。
- 正常的 V2 录制将更加格式正确。
- V1 文件保持现有行为。
- 只有意外依赖 `.acctlm2` 文件末尾 V1 圈索引块的消费者会受影响；这应被视为依赖了一个 bug，而非受支持的契约。

建议测试：

- 录制/写入一个 V2 文件，完成它，运行结束路径，并断言 V2 footer 之后没有出现 V1 `LAP_INDEX_MAGIC`。
- 保留一个 V1 追加回归测试，以证明遗留路径仍然有效。

- 将验证不匹配转为错误：
  - 帧数不匹配应返回 `TelemetryError::InvalidFormat`。
  - 任何校验字段不匹配应返回 `TelemetryError::InvalidFormat`。

可选的兼容性开关：

- 添加 `--allow-mismatch` 用于手动诊断场景。
- 默认行为应保持严格。

预期影响：

- 低到中等。
- 成功的转换不受影响。
- 当前忽略转换器警告的现有脚本，在数据无法完整往返时将开始失败。
- 这是一次行为收紧，但符合转换器的无损契约。

建议测试：

- 添加一个转换器测试，使验证检测到不匹配并断言失败路径。
- 保留现有的成功转换往返测试。

### P2：在 ACC 达到 Live 之前停止被报告为错误

- 位置：
  - `src/recording/engine.rs:121`
  - `src/recording/engine.rs:129`
  - `src/recording/controller.rs:228`
- 症状：
  - `RecordingController::start()` 在线程启动后立即发送 `RecordingStarted`，即使引擎仍在等待 ACC 变为 Live。
  - 如果在写入器创建之前调用 `stop()`，`run_recording_loop()` 返回 `InvalidArgument("stop requested before recording started")`。
  - 控制器将其转换为 `RecordingStatus::Error`，并且不发送 outcome。
- 影响：
  - 在等待 Live 期间用户发起的取消被报告为录制错误。
  - 仅等待 `RecordingOutcome` 的 UI 调用方可能挂起或超时。
  - API 生命周期语义不清晰：`stop()` 被文档描述为结束录制的正常方式，但一个有效的时间窗口被当作无效处理。

#### 修复方案及影响

最小本地修复：

- 将 stop-before-start 当作正常取消而非 `TelemetryError` 来处理。
- 发出诸如 `Stopping { reason: StopReason::Manual }` 的状态，或引入一个 stopped/cancelled 状态。
- 在此情况下不发出 `RecordingStatus::Error`。

更完整的 API 修复：

- 扩展 outcome channel 以区分：
  - 有文件的已完成录制
  - 在任何文件创建之前的取消
  - 录制失败
- 这可以通过诸如 `RecordingCompletion::{Finished(RecordingOutcome), Cancelled, Failed(...)}` 的枚举来完成。

预期影响：

- 最小修复：低到中等。
  - 现有的 outcome 语义保持不变。
  - 调用方必须监听状态才能在文件存在之前观察到取消。
- 完整 API 修复：中等。
  - 更明确、更健壮，但需要下游调用方处理新的完成类型。

建议测试：

- 启动一个保持 `Off` 或 `Unavailable` 的模拟数据源。
- 在任何 Live 帧出现之前调用 `stop()`。
- 断言状态报告的是取消/手动停止，而非错误。
- 断言没有创建输出文件。

### P2：V2 读取器在遇到格式错误的 Row Group/Column 元数据时可能挂起或 panic

- 位置：
  - `src/reader_v2.rs:1178`
  - `src/reader_v2.rs:1299`
  - `src/reader_v2.rs:1326`
  - `src/reader_v2.rs:1350`
- 症状：
  - `scan_row_groups()` 将 `pos` 推进到 `rg_end`。
  - 对于一个有合法 `RGHD` 魔数但没有合法 group end 的格式错误的 row group，`rg_end` 可能与 `pos` 保持不变，导致无限循环。
  - Column 读取使用 `col_data_offset..col_data_offset + byte_len` 进行切片，而不检查末尾是否超出 `group_bytes.len()`。
- 影响：
  - 损坏或恶意的 `.acctlm2` 文件可能导致挂起或 panic，而非返回 `TelemetryError::InvalidFormat`。
  - 这会影响打开或检查用户提供文件的 CLI 操作。
  - 当前默认测试运行未覆盖该风险，因为许多 V2 边缘用例测试仍处于忽略状态。

#### 修复方案及影响

安全的本地修复：

- 在 `scan_row_groups()` 中：
  - 验证 row group 解析有向前推进。
  - 如果计算出的下一个偏移小于或等于当前偏移，返回 `InvalidFormat`。
  - 在信任之前验证每个 `group_end` 是否在 `data.len()` 范围内。
- 在 column 解码路径中：
  - 在切片前检查 `col_data_offset + entry.byte_len as usize <= group_bytes.len()`。
  - 在溢出或超出范围时返回 `TelemetryError::InvalidFormat`。
  - 对偏移加法使用 checked 算术。

预期影响：

- 低。
- 合法的 V2 文件读取应与之前完全一致。
- 无效文件的行为从挂起/panic 变为结构化错误。
- 这使 CLI 行为更安全，而不会改变正常 API。

建议测试：

- 启用或重写 `tests/v2_edge_cases.rs` 中的目标用例。
- 添加以下场景的测试：
  - 零推进 row group
  - 截断的 row group header
  - column 字节长度超出 group end
  - 无效的 skip-index 偏移

### S1：`flush_interval` 配置未在 recording engine 中使用

- 位置：
  - `src/recording/engine.rs`
  - `src/recording/controller.rs:115`
- 症状：
  - `RecordingEngineConfig` 定义了 `flush_interval: Option<Duration>` 字段。
  - `controller.rs:115` 设置了 `flush_interval: Some(Duration::from_secs(2))`。
  - 但 `run_recording_loop()` 的主循环中完全没有检查或使用该配置，writer 的 `flush()` 从未被主动调用。
- 影响：
  - 该配置项被静默忽略，对运行时行为没有任何影响。
  - 数据仅在 `writer.finish()`（录制结束）时 flush 到磁盘。
  - 如果进程崩溃或系统断电，录制期间的所有数据可能丢失。
  - 用户配置的 flush 间隔没有起到预期的数据保护作用。

#### 修复方案及影响

安全的本地修复：

- 在 `run_recording_loop()` 的主循环中加入 `flush_interval` 检查：
  - 在 `RecordingEngineConfig` 旁边维护一个 `last_flush: Instant`。
  - 每次循环 tick 检查 `last_flush.elapsed() >= flush_interval`。
  - 如果条件满足且 `writer` 存在，调用 `writer.flush()` 并更新 `last_flush`。
- 如果 `flush_interval` 为 `None`，保持当前行为（仅在 finish 时 flush）。

预期影响：

- 低。
- 正常录制行为不变，只是增加了磁盘写入频率。
- 崩溃时的数据丢失窗口从"整个录制期间"缩小到"最多一个 flush_interval"。

建议测试：

- 设置 `flush_interval = Some(Duration::from_millis(100))`，录制若干帧后强制终止进程，验证文件已部分写入。
- 设置 `flush_interval = None`，验证仅在 finish 时写入。

### S2：V2 writer 最后一圈 lap entry 的 validity 状态被硬编码

- 位置：

- 症状：

  - 读取 V2 lap index 的工具可能得到错误的 validity / out-lap 信息。
  - 如果 future tools 依赖 V2 footer 的 lap metadata 而不是重新扫描文件，会得到错误结果。

#### 修复方案及影响

安全的本地修复：

- 在 `BinaryTelemetryWriterV2` 中增加字段跟踪当前圈的 `is_valid: bool` 和 `is_out_lap: bool`。
- 在 `write_frame()` 中根据 session 数据更新这些状态。
- 在 `finish()` 中使用实际状态而不是硬编码。

预期影响：

- 低。
- 只有 V2 footer 中的 lap metadata 会改变，实际数据不受影响。
- 需要确认是否有下游工具依赖当前硬编码的行为（应该没有）。

建议测试：

### S3：`append_lap_index` 读取整个 session cluster 可能导致 OOM

- 位置：
  - `src/recording/outcome.rs:363`
- 症状：
  - `append_lap_index()` 调用 `BinaryTelemetryReader::open(path)` 然后 `read_all_session()`。
  - `read_all_session()` 会将整个文件的 session cluster 全部读入内存。
- 影响：
  - 对于长时间录制的文件（如数小时的 endurance race），session cluster 可能非常大。
  - 可能导致内存占用过高甚至 OOM。
  - 由于 `append_lap_index` 在录制结束后调用，此时用户可能正在进行其他操作，突然的内存压力会影响用户体验。

#### 修复方案及影响

安全的本地修复：

- 使用流式读取代替 `read_all_session()`：
  - 只读取 `sample_tick` 和 `normalized_car_position` 两个字段。
  - 逐 chunk 读取，而不是一次性读入内存。
- 或者，在 `BinaryTelemetryReader` 上增加一个只读取指定字段的 API。

替代方案：

- 完全避免 `append_lap_index` 对 V2 文件的调用（修复 P1 后这个问题自动消失）。
- 对于仍然需要 V1 lap-index 的场景，可以限制文件大小或增加内存警告。

预期影响：

- 低。
- 正常大小的文件不受影响。
- 大型文件的内存占用会显著降低。

建议测试：

- 用一个超大文件测试 `append_lap_index()` 的内存占用。
- 验证流式读取的 lap boundary 检测结果与 `read_all_session()` 一致。
