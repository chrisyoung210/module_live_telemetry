## 审计发现

### P1：DeltaTimeToLifeBestLap 时间差符号与其他 delta 项相反

- 位置：
  - `src/compute/items.rs:307`
- 症状：
  - `DeltaTimeToLifeBestLap::compute()` 返回 `ref_time - current_time`。
  - 当当前圈比参考圈慢时（`current_time > ref_time`），返回值为**负数**。
  - 而所有其他 delta 项返回 `current_time - reference_time`，慢时为**正数**：
    - `DeltaTimeToLifeBestLapInterpolated`：`items.rs:429` — `current_time_ms - reference_time_ms`
    - `DeltaTimeToSessionBestLap`：`items.rs:537` — `current_time - ref_time`
    - `DeltaTimeToSessionBestLapInterpolated`：`items.rs:667` — `current_time_ms - reference_time_ms`
  - `DeltaTimeToLifeBestLap` 的文档注释（`items.rs:249-252`）声明"正值 = 比参考圈慢"，与代码矛盾。
  - 测试 `test_delta_time_to_life_best_lap_normal`（`items.rs:1704`）断言 `delta ≈ -1000`（当前圈慢 1000ms），确认了反转的符号约定。
- 影响：
  - 如果 dashboard 消费者同时订阅 `delta_time_to_life_best_lap` 和 `delta_time_to_session_best_lap`，同一语义（当前圈落后）会显示为相反符号。
  - HUD 显示可能出现"+1.2s"和"-1.2s"交替出现的混乱。
  - 依赖符号判断快慢的逻辑（如"delta > 0 表示落后"）会对 `delta_time_to_life_best_lap` 产生错误判断。

#### 修复方案及影响

确认方案（无歧义）：

1. **符号翻转**：`src/compute/items.rs:307`，将

   ```rust
   return Ok(ref_time - current_time);
   ```

   改为

   ```rust
   return Ok(current_time - ref_time);
   ```

   使其与 `DeltaTimeToLifeBestLapInterpolated`（`items.rs:429` `current_time_ms - reference_time_ms`）、`DeltaTimeToSessionBestLap`（`items.rs:537`）、`DeltaTimeToSessionBestLapInterpolated`（`items.rs:667`）一致：**正值 = 比参考圈慢**。

2. **测试断言更新**（仅翻转符号，测试数据本身不变）：

   - `test_delta_time_to_life_best_lap_normal`（`items.rs:1704`）：当前圈慢 1000ms，修复后 delta ≈ +1000。

     ```rust
     // 旧：assert!((delta + 1000.0).abs() < 1.0);
     assert!((delta - 1000.0).abs() < 1.0);
     ```

   - `test_delta_time_to_life_best_lap_reset_on_lap_change`（`items.rs:1749`）：当前圈慢 2000ms。

     ```rust
     // 旧：assert!((delta + 2000.0).abs() < 1.0);
     assert!((delta - 2000.0).abs() < 1.0);
     ```

3. **新增对比测试** `test_delta_time_to_life_best_lap_consistent_sign_with_interpolated`：用同一组参考圈与当前帧分别调用 `DeltaTimeToLifeBestLap` 与 `DeltaTimeToLifeBestLapInterpolated`，断言两者返回值**同号**且差的绝对值 < 10ms（Interpolated 做 round 到 10ms，允许少量差异）。参考圈不含 out-lap（避免与 P1-2 裁剪逻辑耦合）。

预期影响：

- 低到中等。
- 当前仓库内除测试外无其他处依赖该符号（消费方为 dashboard sink 输出，仅透传数值）。
- 修复后与所有其他 delta 项符号约定统一。

### P1：DeltaTimeToLifeBestLap 未裁剪参考圈到 S/F 穿越点

- 位置：
  - `src/compute/items.rs:281-314`
- 症状：
  - `DeltaTimeToLifeBestLap::compute()` 直接从 `reference[0]` 开始扫描，不检测 S/F 穿越点。
  - 如果参考圈文件包含 out-lap 数据（S/F 穿越前的帧），这些帧的 `i_current_time` 属于上一圈，`normalized_car_position` 从 0.0 开始而非 S/F 后的值。
  - 所有其他 delta 项都检测 S/F 穿越并裁剪参考圈：
    - `DeltaTimeToLifeBestLapInterpolated::refresh_reference_points()`：`items.rs:357-366` — 使用 `next_back()` 查找最后一次 S/F 穿越。
    - `DeltaTimeToSessionBestLap::compute()`：`items.rs:484-504` — 查找 S/F 穿越。
    - `DeltaTimeToSessionBestLapInterpolated::refresh_reference_points()`：`items.rs:594-603`。
    - `PredictLapTimeBase::refresh_reference_total_time()`：`items.rs:718-726`。
    - `DeltaTimeToSessionBestRefPoly::refresh()`：`items.rs:946` — 使用 `find_sf_start()`。
- 影响：
  - 当参考圈文件包含 out-lap 数据时，delta 计算会使用错误的时间基准和错误的位置匹配。
  - `i_current_time` 在 S/F 穿越时重置为 0，但 out-lap 帧的时间属于上一圈，导致 `ref_time - current_time` 计算出错误的时间差。
  - `normalized_car_position` 在 out-lap 中可能从 0.0 开始递增，与飞驰圈的 S/F 后位置序列不一致，导致线性扫描匹配到错误区间。

#### 修复方案及影响

确认方案：在 `DeltaTimeToLifeBestLap::compute()`（`items.rs:281-314`）中，于 `reference.is_empty()` 检查之后、`let current_lap = ...` 之前，插入 S/F 穿越裁剪，采用与 `DeltaTimeToLifeBestLapInterpolated::refresh_reference_points()`（`items.rs:357-366`）完全相同的判定：

```rust
let reference = ctx.reference_lap.ok_or(ComputeError::NoValidData)?;
if reference.is_empty() {
    return Err(ComputeError::InvalidReferenceData);
}

// 裁剪到最后一次 S/F 穿越之后（丢弃 out-lap 帧）
let start = reference
    .windows(2)
    .enumerate()
    .filter(|(_, pair)| {
        pair[1].session.normalized_car_position + 0.5
            < pair[0].session.normalized_car_position
    })
    .map(|(i, _)| i + 1)
    .next_back()
    .unwrap_or(0);
let reference = &reference[start..];
if reference.is_empty() {
    return Err(ComputeError::InvalidReferenceData);
}

let current_lap = ctx.current_frame.session.completed_laps;
// ...（后续逻辑不变）
```

注意事项：

- **`self.index` 语义**：`self.index` 是对**裁剪后** `reference` 切片的游标。同一参考圈的 `start` 每次调用都相同，且 `self.index` 在圈变更时重置为 0（`items.rs:293`），因此裁剪后游标始终有效。已有的边界检查 `if self.index < reference.len()`（`items.rs:295`）针对的是裁剪后的 `reference`，**无需改动**。
- **不含 out-lap 的参考圈**：`start = 0`，`reference` 不变，行为完全不变。
- **判定阈值 `+ 0.5`**：必须与 Interpolated 变体一致（检测 `normalized_car_position` 从 ~1.0 回落到 ~0.0 的 S/F 穿越）。不要改成其他阈值，否则两变体可能裁剪到不同起点。
- **`next_back()` 而非 `.last()`**：这里用 `next_back()`（`DoubleEndedIterator`），与 Interpolated 变体一致；同时消除 clippy `double_ended_iterator_last` 告警（见 P2 clippy 表 `items.rs:484/594/718`，但本处是新增代码，保持与 Interpolated 一致即可）。

预期影响：

- 低。不含 out-lap 的参考圈不受影响（`start = 0`）。
- 含 out-lap 的参考圈将正确忽略 S/F 前的数据，`i_current_time` 与 `normalized_car_position` 基准与飞驰圈一致。

建议测试：

- 新增 `test_delta_time_to_life_best_lap_trims_out_lap`：构造参考圈，前段为 out-lap（`i_current_time` 从上一圈延续、`normalized_car_position` 从 0.0 递增到 ~1.0 后回落到 0.0），后段为飞驰圈（`i_current_time` 从 0 开始）。验证 `compute()` 使用飞驰圈时间基准——与 `DeltaTimeToLifeBestLapInterpolated` 在相同输入下返回接近的 delta。

### P2：LapCompletedEvent.is_out_lap 被硬编码为 false

- 位置：
  - `src/recording/engine.rs:298`
  - `src/recording/engine.rs:418`（replay loop）
- 症状：
  - `run_recording_loop()` 在检测到圈完成时，构造 `LapCompletedEvent::new(...)` 的第 4 个参数 `is_out_lap` 硬编码为 `false`。
  - `run_replay_loop()` 同样硬编码为 `false`（`engine.rs:418`）。
  - 基线提交之前的代码计算 `is_out_lap = lap_completed_count == 0`，但此逻辑在重构中被移除。
  - `SessionBestLapTracker::consider_lap()`（`controller.rs:720-725`）检查 `event.is_out_lap` 并跳过 out-lap。由于 `is_out_lap` 永远为 `false`，out-lap 永远不会被跳过。
- 影响：
  - Out-lap（从维修区驶出的圈，通常较慢）会被错误地纳入 session best 候选。
  - 如果 out-lap 的 `lap_time_ms` 恰好小于已有最佳圈（在极端情况下可能发生），session best 参考圈会被设置为 out-lap 数据。
  - 这会导致后续所有 delta-time 计算基于一个不代表真实飞驰圈速度的参考圈，产生系统性偏差。

#### 修复方案及影响

> **勘误**：本节早期版本建议 `is_out_lap: completed_laps == 0`，这是**错误的**。`LapCompletedEvent` 在 `completed_laps` **递增后**触发（`engine.rs:290` 条件 `completed_laps > previous`），事件携带的是递增后的值——out-lap 完成时 `completed_laps` 从 0 变为 1，事件拿到的是 `1`，`== 0` 永不成立。正确做法见下。

确认方案：恢复基线提交 `e6b9400` 的**内部计数器**逻辑（不依赖 ACC `completed_laps` 字段值，避免 off-by-one）。

基线原文（`git show e6b9400:src/recording/engine.rs`）：

```rust
let mut lap_completed_count: u32 = 0;
...
if completed_laps > last_completed_laps {
    let is_out_lap = lap_completed_count == 0;   // 第一个完成事件 = out-lap
    ...
    cb(LapCompletedEvent::new(
        lap_completed_count,    // 旧代码用内部计数器作 lap_number（0-based）
        is_valid, lap_time_ms, is_out_lap, lap_frames,
    ));
    lap_completed_count += 1;
}
```

当前重构版把 `last_completed_laps` 改为 `Option<u32>`、`lap_number` 改为 ACC 的 `completed_laps`。**本次只恢复 `is_out_lap`，不回退 `lap_number` 语义**（`lap_number` 改动不在本审计范围，下游仅用于日志）。

改动两处，结构相同：

**A. `run_recording_loop()`**（事件构造 `engine.rs:294-303`，状态声明在其上方函数体起始处、与 `last_completed_laps` 同一区域）：

- 在 `last_completed_laps` 声明旁新增：

  ```rust
  let mut lap_completed_count: u32 = 0;
  ```

- `LapCompletedEvent::new(...)` 的第 4 参数从硬编码 `false` 改为：

  ```rust
  lap_completed_count == 0,
  ```

- 在 `cb(...)` 调用**之后**、`current_lap_is_valid = ...` 之前，新增：

  ```rust
  lap_completed_count += 1;
  ```

**B. `run_replay_loop()`**（事件构造 `engine.rs:460-467`，状态声明 `engine.rs:406-408`）：完全相同的改动（注释 `// Lap completion detection state (matches run_recording_loop)` 已表明两处应保持一致）。

一致性核对：

- 与 `writer_v2.rs:562` `self.current_lap_is_out_lap = lap == 1`（其中 `lap = completed_laps + 1`，即 `completed_laps == 0` 时为 out-lap）**语义一致**：两者都把"第一条圈"标记为 out-lap。区别仅在于 writer_v2 在圈**开始**时判定，LapCompletedEvent 在圈**完成**时判定——内部计数器 `lap_completed_count == 0` 正好对应"第一个完成事件"，等价。

预期影响：

- 低。
- `SessionBestLapTracker::consider_lap()`（`controller.rs:720-725`）将正确跳过 out-lap，仅考虑飞驰圈作为 session best 候选。
- 不影响 `lap_number` 字段语义（仍为 ACC 的 `completed_laps`）。

建议测试：

- 新增/扩展 `engine.rs` 测试模块的 lap 完成测试：构造 `ScriptedTelemetrySource` 序列，使 `completed_laps` 经历 0→1（out-lap 完成）→2（飞驰圈完成）。断言第一个 `LapCompletedEvent.is_out_lap == true`、`lap_number == 1`；第二个 `is_out_lap == false`。
- 扩展 `controller.rs` 的 `SessionBestLapTracker` 测试：喂入一个 out-lap 事件（`is_out_lap == true`、`lap_time_ms` 故意很小）和一个飞驰圈事件，断言 tracker 选择了飞驰圈而非 out-lap。
